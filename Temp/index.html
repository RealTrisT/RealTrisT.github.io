Bootloader e “kernel” de teste para x86

https://github.com/RealTrisT/OS_DEV

Projeto que corre barebones em x86. Os processadores de x86 todos são iniciados com modo de compatibilidade para 16 bits, e é preciso uma quantidade razoável de inicialização para:

  • Trocar para do modo Real para modo Protected de 32 bits
  • Utilizar um ecrâ - apenas possível de forma razoável com VBE (vesa bios extensions) que é o equivalente a uma driver presente na BIOS de uma motherboard que tenha o seu próprio conector e gráficos (a maioria das modernas têm).
  • Utilizar input (teclado e rato), que apenas é possível de forma prática com comunicação serial (que era o usado pelos teclados PS/2 com o cabo redondo verde)

Aprendi bastante, mas infelizmente não seria nada prático tentar fazer um sistema operativo (que era a ideia inicial) pois é preciso fazer drivers manualmente para tudo (gráficos, USB, SATA, etc…) e a documentação para SATA é paga, a documentação para USB é tão enorme e bloated que seria extremamente difícil de implementar em tempo útil, e a interface com as placas gráficas são completamente proprietárias (ou seja impossível fazer drivers para uma placa da Nvidia ou AMD sem reverse engineering).

O resultado final foi um ecrâ onde podia escrever com um teclado PS/2 emulado e executar comandos. Podia ter feito um MS-DOS meu, mas imagino que drivers de rede fossem igualmente dolorosas de implementar e por isso não haveria muito uso, preferível fazer um kernel/sistema operativo para ARM ou outra arquitetura onde o meio não o torne impossível.


código da parte do bootloader que escolhe o modo de VBE (funcionalidade inacessível fora de Real Mode)

Network Sniffer

Não tenho provas de que alguma vez realizei este projeto.
Um programa que vê o tráfego na rede (que é visível ao sistema).
Implementei parsers para os protocolos onde a maior parte do tráfego é feito (TCP e UDP), bem como protocolos de controlo (ARP, ICMP), e obviamente os de mais baixo nível que transportam estes (Ethernet, IPv4 e IPv6).
Fiz também uma driver para o linux onde dei hook a uma das funções de networking do kernel com o intuito de transmitir por IOCTL os packets ao processo de usermode (antes de serem filtrados pelo kernel, dado que isto fazia com que os raw sockets apenas apresentassem os packets direcionados para o mac address do sistema). Raw sockets com monitor mode dão os pacotes com os headers de IEEE 802.11 (penso eu, infelizmente nunca cheguei a mexer com wifi - mas era claro que os pacotes não eram ethernet frames), e por isso pareceu a maneira mais fácil.

Porém ao fim de muitos kernel panics quando consegui fazer a hook (a minha primeira hook em 64 bits) não fiquei com vontade de continuar imediatamente para a implementação dos IOCTLs. Meses depois quando voltei ao projeto - o standard do c++ tinha dado update, e todos os const casts implícitos, accessos a ponteiros void e arrays de tamanho 0 que eu tinha estavam incompiláveis. O programa passou a ter de um dia para o outro milhares de erros.

Enervado, eliminei o projeto completamente, e recomecei uma versão que nunca chegou ao mesmo patamar.
https://github.com/RealTrisT/NetworkSniffer.
A esse ponto teria que rescrever quantidades enormes de código sem aprender nada e perdi o interesse, ferramentas que fazem o mesmo já existem e por isso não havia objetivo.

Sistema de hooks em processos externos

https://github.com/RealTrisT/LoudHooker

Este projeto é um DLL que quando injetado num processo permite fazer até 4 hooks em partes diferentes do programa, e interpretar as registers ou memória do stack como diferentes tipos de dados, mostrando-as numa interface web.

Programei também para este projeto um mini servidor de websockets que serve para transmitir as informações das chamadas para a interface. Bem como uns sockets que servem uma página web (http headers hardocded).


Video que mostra a funcionalidade deste programa.

O nome não é o mais profissional, mas não considerei usar os meus projetos como reportório até mais tarde. Mas “Loud” porque o método usado é o uso das debug registers do x86, e tudo o que um processo alvo precisa de fazer para ver se está sobre o “efeito” deste programa é ver se as registers estão ativas. O resto do nome é só devido ao facto de o programa fazer hooks.

Injetor

Um injetor é uma peça de software que injeta código dentro de outro processo. Isto é o que seria utilizado por exemplo em conjunção com o LoudHooker, pois o mesmo é uma biblioteca que precisa de estar dentro do processo alvo para conseguir parar a execução e olhar para as registers e stack. Injeção pode ser utilizado por exemplo também para adicionar funcionalidade a um processo, ou para o loading de um plugin em situações um pouco mais raras.

https://github.com/RealTrisT/Injectah

Este é mais como uma menção honorável. É um dos meus projetos mais antigos e o início da minha jornada de baixo nível.
Este injetor é porém extensível, vários métodos de injeção mais sofisticados podem ser adicionados ao programa, o propósito principal é a facilidade de uso, que ao fim de vários meses a injetar DLLs se torna importante.

Gestores de projetos

O github é uma boa ferramenta mas para desenvolvimento individual é mais trabalhoso do que eu gostaria e não trata dos ambientes pessoais. A ideia é um conjunto de ferramentas que com um comando abram os programas necessários e deixem o ambiente pronto para continuar onde se deixou por último.


Lista com projetos em progresso, descrições e as suas localizações.
Em conjunto com este comando há um openproject que abre o ambiente para o projeto dado como argumento por nome. Isto é útil pois os projetos acumulam e mesmo com uma pasta dedicada a projetos pode demorar uns segundos a encontrar o que se procura (o windows não é consistente com ordenação e os sistemas de ficheiros normalmente não dão para ordenar de forma específica). Assim sabendo o nome de um projeto nem precisa de se procurar em qualquer lista, basta usar o comando openproject, e não se lembrando do nome de um projeto pode-se chegar lá pela descrição.


Programas a inicializar através dos projetos (ultimamente só tenho programado C, python e web - apenas utilizo estes).


Lista dos projetos bem como informação sobre que processos abrir e com que argumentos.

Eventualmente este path seria transformado num “SyncPath” que conteria uma hash do ID do volume ao qual os ficheiros pertencem. Poder-se-ia assim colocar vários SyncPaths para os diferentes sistemas ou discos, sendo implementadas depois file hooks e sincronizações para atingir essa funcionalidade final de sincronização.
